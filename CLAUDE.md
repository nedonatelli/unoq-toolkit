This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

unoq-toolkit: LED Matrix Animator and tools for Arduino UNO Q (STM32U585 Cortex-M33, Zephyr RTOS). The board has a 13x8 LED matrix (104 LEDs) and a Qualcomm QRB2210 Linux SoC connected via serial UART.

## Commands

```bash
# Lint
ruff check .

# Build docs (warnings are errors in CI)
cd docs && make html

# Compile a sketch
arduino-cli compile --fqbn arduino:zephyr:unoq <sketch_dir>

# Upload a sketch
arduino-cli upload -p /dev/cu.usbmodem19087929472 --fqbn arduino:zephyr:unoq <sketch_dir>

# Run the GUI
python gui.py

# Push and run a board script via ADB
ADB=~/Library/Arduino15/packages/arduino/tools/adb/32.0.0/adb
$ADB push board/clock.py /tmp/clock.py
$ADB shell "python3 /tmp/clock.py"
```

## Architecture

### Host-side (macOS)

- **gui.py** — Tkinter GUI for designing LED patterns. Contains shape/animation presets and live sketches in `SHAPES`, `ANIMATIONS`, and `LIVE_SKETCHES` dicts. Generates `.ino` files into `blink/blink.ino` and compiles/uploads via `arduino-cli` in a background thread.
- **wifi_scan.py** — Scans WiFi via ADB, generates a static sketch with signal bars, compiles and uploads each cycle.

### Board-side (Linux on UNO Q)

- **board/clock.py** — Renders HH:MM with blinking colon using a 3x5 pixel font, pushes frames via Bridge RPC every 0.5s.
- **board/wifi_monitor.py** — Scans WiFi via `nmcli`, builds bar chart frames, pushes via Bridge RPC every 1s.
- Both use `SimpleBridge` class connecting to `/var/run/arduino-router.sock` (Unix domain socket).

### MCU sketches

- **blink/** — Scratch directory, auto-generated by gui.py on each upload. Do not edit manually.
- **rpc_receiver/** — Bridge RPC receiver sketch. Registers `set_frame()` method so Linux-side scripts can push frames.
- **joystick/** — Joystick-controlled 2x2 square on the LED matrix. VRX→A1, VRY→A0 with auto-calibration.
- **runner/** — Side-scroller jump game on the LED matrix with joystick input.
- **tft_hwspi/** — Spinning 3D wireframe cube on TFT display. Hardware SPI at 60MHz, fixed-point math, partial canvas (160x120). Requires `Arduino_ST7789.cpp` SPI_MODE0 patch.
- **tft_swspi/** — Same spinning cube using software SPI. No library patch needed, but slower (longer lines = lower FPS).

## LED Matrix Frame Format

13 columns × 8 rows = 104 bits packed into 4× `uint32_t`, MSB-first, row-major:
```python
i = row * 13 + col
frame[i // 32] |= (1 << (31 - (i % 32)))
```

## Bridge RPC (arduino-router)

The arduino-router daemon owns `/dev/ttyHS1` exclusively. Scripts on the Linux side must communicate through it, not directly to the serial port.

**MCU side** — Use `Arduino_RouterBridge.h` (NOT `Arduino_RPClite.h`):
```cpp
#include <Arduino_RouterBridge.h>
Bridge.begin();
Bridge.provide("method_name", handler_func);  // registers with router
```

**Linux side** — Connect to Unix socket, send MsgPack-RPC `[0, msgid, method, [params]]`:
```python
bridge = SimpleBridge()
bridge.connect()  # connects to /var/run/arduino-router.sock
bridge.call("set_frame", hex_frame_string)
```

`Arduino_RPClite` is for direct serial — it does NOT register methods with the router and calls will fail with "method not available."

## TFT Display (Adafruit Mini PiTFT 135x240)

ST7789 driver, 240x135 landscape. Hardware SPI works after patching `Arduino_ST7789.cpp` to use `SPI_MODE0` (the library defaults to `SPI_MODE2` for unknown platforms).

Wiring:
| Display | UNO Q |
|---------|-------|
| SCK     | D13   |
| MOSI    | D11   |
| CS      | D10   |
| DC      | D9    |
| RST     | D8    |
| BL      | D7    |
| VIN     | 3.3V  |
| GND     | GND   |

Library: `GFX Library for Arduino` (Arduino_GFX). The Adafruit ST7789 library does not compile on Zephyr (missing `pins_arduino.h`).

ST7789 GRAM offsets for 135x240 panel: `col_offset1=52, row_offset1=40, col_offset2=53, row_offset2=40`.

### Hardware SPI patch

The Arduino_GFX library defaults to `SPI_MODE2` for unknown platforms in `Arduino_ST7789.cpp`. The ST7789 requires `SPI_MODE0`. Patch the installed library:
```
~/Documents/Arduino/libraries/GFX_Library_for_Arduino/src/display/Arduino_ST7789.cpp
```
Change `SPI_MODE2` → `SPI_MODE0` in the `begin()` method. Reinstalling the library will overwrite this patch.

### Performance notes

- HW SPI at 60MHz is the fastest reliable clock for this board + display combo.
- Full-screen framebuffer (240x135, 64KB) is too slow even with HW SPI. Use a partial canvas (160x120) centered on the render area.
- Fixed-point math (10-bit, 256-entry sine LUT) eliminates all floating-point from the render loop.
- SW SPI works without patches but is ~10-50x slower — use only as a fallback.

## Board Quirks

- `Serial` maps to hardware UART pins (usart1), NOT USB CDC. USB port is programming-only.
- Bootloader recovery: `arduino-cli burn-bootloader --fqbn arduino:zephyr:unoq -p PORT -P openocd`
- Default `wait_linux_boot=yes`; use `wait_linux_boot=no` for standalone sketches.
